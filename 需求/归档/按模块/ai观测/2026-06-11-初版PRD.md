# 产品需求文档 (PRD)：LLM AI观测 (LLM Observability)

## 1. 文档概述
### 1.1 项目背景
随着大语言模型（LLM）应用的爆发，传统的 APM（应用性能监控）和日志工具已无法满足 AI 应用非确定性、长文本、高成本、多轮交互的监控需求。我们需要一款专为 LLM 应用设计的可观测性（Observability）平台，帮助开发者实现全链路追踪、成本核算、性能调优及数据合规。

### 1.2 产品目标
打造一套高可用、低延迟、易集成的 LLM 可观测性服务。核心目标包括：
1. **全链路透视**：清晰展示 LLM 应用从输入到输出的完整执行链路（Trace）。
2. **精细化成本管控**：精确追踪 Token 消耗与财务成本。
3. **性能瓶颈定位**：监控延迟，特别是流式输出的首字响应时间（TTFT）。
4. **安全与合规**：提供 PII（个人敏感信息）脱敏能力，保障数据隐私。

### 1.3 目标用户
- **AI 研发工程师**：调试 Agent，排查 Agent 工作流错误，分析延迟。
- **产品经理**：追踪用户行为，分析 Session 会话质量，统计业务指标。
- **运维/安全人员**：监控系统负载，管理数据保留策略，配置脱敏规则。

## 2. 核心数据模型设计 (Data Model)
开发团队需基于以下核心实体设计底层存储（建议采用 ClickHouse 等 OLAP 数据库以支持海量 Trace 查询，结合 PostgreSQL 存储元数据）。

| 实体名称 | 描述 | 核心属性 |
| :--- | :--- | :--- |
| **Workspace** | 隔离的业务工作空间 | ID, Name, API Keys (Public/Secret) |
| **Session** | 会话/工作流聚合组 | ID, Workspace ID, 包含的 Trace IDs |
| **Trace** | 一次完整的请求生命周期 | ID, Session ID, User ID, Name, Environment, Tags, Metadata, Input, Output, Timestamp |
| **Observation** | Trace 内的具体执行节点 | ID, Trace ID, Parent ID, **Type**, Name, Start/End Time, Input, Output, Model, Tokens, Cost, Metadata |

**Observation 类型枚举 (Type)：**
`GENERATION` (LLM生成), `SPAN` (普通耗时操作), `EVENT` (瞬时事件), `AGENT`, `TOOL`, `CHAIN`, `RETRIEVER`, `EVALUATOR`, `EMBEDDING`, `GUARDRAIL`。

## 4. 功能说明

本章对第 3 章各功能点给出交互规则、业务逻辑与验收标准；**子模块编号与 §3 对齐**（§4.1 对应 §3.1，以此类推），功能序号 `F-xxx` 与 §3 一致。

### 4.1 Dashboard（仪表盘）

#### F-101 关键指标卡片
- **指标**：Traces、Sessions、Total Cost、Avg Latency、Error Rate，共 5 张卡片。
- **规则**：每张卡片展示当前值 + 环比趋势（如「+12% vs 上周」）；趋势颜色：指标变好为绿，变差为红（Error Rate 反向）。
- **验收**：5 项指标均有数值与趋势文案；数据来自服务端聚合或本期 mock。

#### F-102 指标趋势分析
- **展示**：支持 7 天、30 天等时间粒度的成本、延迟、调用量趋势图（柱状图 / 折线图）；原型默认展示 7 天 Trace 量柱状图。
- **规则**：X 轴为日期，Y 轴为相对高度或绝对数值；切换时间粒度后重新请求聚合 API。
- **验收**：7 天视图下柱体与日期标签一一对应；无数据日期显示 0 或空柱。

#### F-103 Token & 成本细分
- **字段**：今日成本、P95 延迟、总 Tokens、TTFT P50 等细分聚合指标。
- **规则**：与 KPI 卡片同屏或置于次级卡片区域；数值单位分别为 $、ms、M tokens、ms。
- **验收**：四项同屏展示；单位正确。

#### F-104 模型用量排行
- **表格列**：Model、Traces、Tokens、Cost ($)、Avg Latency。
- **规则**：按 Traces 或 Cost 降序（默认 Traces）；支持多模型对比。
- **验收**：至少展示 3 个模型行；Model 名为等宽字体。

#### F-105 看板维度自定义
- **本期**：文案占位「支持按 Workspace、Model、Tag 快速切换和自定义视图」。
- **后续**：用户可勾选维度、拖拽卡片排序并保存个人/Workspace 级布局。

#### F-106 AI 智能问答
- **交互**：页顶文本框输入中文问题 →「智能分析」生成解读；「清除」清空输入与结果。
- **数据范围**：仅聚合当前用户**有权限**的 Workspace 数据；文案中明示当前 Workspace 名称。
- **输出**：标题 + 正文（支持简单 Markdown 渲染）；可基于 mock 规则返回成本、延迟、错误来源等摘要。
- **验收**：输入「最近错误主要来自哪里」类问题可得到结构化回复；清除后输入框与结果区为空。

- **TTFT**：仅 `GENERATION` 类型展示；依赖 SDK 上报 `ttftMs` 或由 `completion_start_time` 与 `start_time` 计算。

- **规则**：Type 列显示类型前两字母缩写 + 色彩 class（`obs-type GENERATION` 等）；悬停 title 展示完整类型名。
- **验收**：7 种类型视觉可区分。

- **Latency**：`latencyMs / 1000`，保留 2 位小数 + 单位 s；空值显示「—」。

- **验收**：折叠后面板不占横向空间；角标与条件数一致。

#### F-307 Workspace 下拉
- **选项**：`All Workspaces` + `SIDEBAR_CONFIG.workspace.options` 中的各 Workspace。
- **规则**：变更即触发列表刷新；与 Filters 面板内条件 AND 组合。
- **验收**：选择「人工智能实验室」仅显示该 workspace 字段匹配的行。

- **模式**：`IDs / Names`（匹配 id、name、traceName、traceId、userId）；`All fields`（额外含 input、output 等）。
- **规则**：与 Filters 条件 AND；输入后实时或回车触发过滤（原型为即时）。
- **验收**：搜索 `tr_8f3a` 可定位对应 Trace 相关行。

#### F-309 时间范围
- **选项**：至少 `1d Past 1 day`、`7d Past 7 days`；作用于列表查询 time range 参数。
- **验收**：切换后列表数据范围变化（本期可为 mock 过滤）。

#### F-310 分页
- **控件**：Rows per page（50/100）、当前页/总页、首页/上一页/下一页/末页。
- **规则**：默认 50；总条数为过滤后计数。
- **验收**：底栏展示 `Page X of Y · N rows`；首页时向前按钮 disabled。

- **入口**：工具栏「AI 观测」按钮，展开/收起面板。
- **能力**：「概览分析」输出当前筛选集成功/失败统计；「追问」基于上次结果继续问答。

#### F-401 分组折叠筛选
- **分组**：Type（多选）、Is Root Observation（true/false）、Trace Name / Name / Model（文本）、Trace Tags（多选）、User ID / Session ID / Trace ID（文本）、Status（ok/error）。
- **交互**：点击分组标题展开/折叠；默认展开 type、name、tags（可配置）。

- **验收**：移除 Type 中某一项后列表即时更新。

#### F-403 Clear all
- **范围**：清空 Filters 面板内全部字段；**不**重置 Workspace 工具栏下拉、**不**关闭 AI 筛选输入框（可选择一并清空 AI 筛选文案，原型实现为重置 `traceFilters` 至 empty）。
- **验收**：点击后 Filters 角标为 0，列表恢复仅受 Workspace/搜索约束。

#### F-404 AI 筛选
- **交互**：Filters 头部「AI 筛选」展开中文 textarea →「应用筛选」解析自然语言 → 回显已应用规则摘要。
- **解析示例**：「错误」→ Status=error；「gpt-4」→ Model 含 gpt-4o；「用户 2847」→ userId；「对话」→ tags feature:chat + traceName 含 chat。
- **规则**：解析结果写入 `traceFilters`，用户可在各分组继续手工修改；应用后显示 `obs-ai-filter-badge`。
- **验收**：输入「显示错误的 tool 调用」后 Type=TOOL 且 Status=error（或等价规则）生效。

### 4.4 Trace 详情抽屉

#### F-501 右侧抽屉 + 遮罩
- **尺寸**：宽度约主内容区 72%；`top` 对齐全局顶栏下沿；左侧列表仍可见。
- **关闭**：点击遮罩、头部 ✕、按 Esc（若绑定）关闭抽屉并取消行高亮。
- **验收**：抽屉不遮挡 Sidebar 与顶栏。

#### F-502 Observation 树
- **结构**：按 parent-child 递归渲染；根节点无 parent；展示 type 缩写、name、duration、cost（若有）。
- **交互**：点击节点更新右侧 Preview 区为对应 Observation 的 input/output。
- **验收**：多层级 Trace 树缩进正确；选中节点有 `active` 样式。

#### F-503 树内搜索
- **规则**：过滤 name 或 type 包含关键字的节点；若子节点匹配则保留父路径。
- **验收**：输入 `llm` 仅显示名称匹配的节点及其祖先。

#### F-504 Topology 拓扑
- **展示**：左下方面板「Topology」，以节点+连线展示 AGENT/TOOL/CHAIN 调用关系。
- **联动**：点击拓扑节点与树节点、右侧详情同步选中。
- **验收**：含 tool-call 的 Trace 可见拓扑结构；无子节点时显示叶子节点。

#### F-505 元数据徽章
- **字段**：Timestamp、Latency、Session（可点击链接占位）、User（可点击链接占位）、Workspace、Total Cost、Tokens（prompt → completion → total）。
- **验收**：与当前 Trace 聚合数据一致；Session/User 链接本期可为 `#`。

#### F-506 Preview 标签
- **内容**：Input、Output 分区；JSON 字符串原样展示在 `<pre>`；各块带复制按钮。
- **Token**：GENERATION 节点额外展示 Token Usage JSON。
- **验收**：切换树节点后 Input/Output 随节点变化；复制点击有反馈（原型可为 alert）。

#### F-509 / F-510 导航与全屏
- **本期**：上一条/下一条、全屏按钮 disabled 或占位。
- **后续**：上一条/下一条在 Tracing 列表当前排序结果集中切换 traceId；全屏隐藏左侧列表仅展示详情。

### 4.5 后端与数据依赖

#### F-601 Observation 上报
- SDK 异步批量上报；单条 Observation 须含 §2 数据模型字段；`workspace` 由 API Key 或显式参数绑定。
- **验收**：上报后 5s 内在 Tracing 列表可查到（P99）。

#### F-602 Workspace 隔离查询
- 查询 API 须校验调用方对 Workspace 的读权限；不得跨租户泄露数据。
- **验收**：用户 A 无法通过改参数查询用户 B 的 Workspace Trace。

#### F-603 列表查询 API
- 支持 §3.3 全部筛选字段 + §3.2 搜索模式 + 分页 + Start Time 排序。
- **验收**：组合 3 个以上筛选条件时 P95 响应 < 2s（千万级数据量目标）。

#### F-604 Trace 详情 API
- 输入 `trace_id`，返回 observations 数组（含 parent_id、children）、聚合 tokens/cost、tags。
- **验收**：详情树节点数与上报一致，无孤儿节点。

#### F-605 成本与 Token 计算
- 服务端维护 Model 单价表；GENERATION/EMBEDDING 类型自动计算 cost 与 token 分项。
- **验收**：详情抽屉 Total Cost 与列表聚合误差 < 0.01%。

#### F-606 AI 分析服务
- 独立服务或 LLM 网关；输入自然语言 + 当前筛选上下文 → 输出分析文本或结构化 filters。
- **安全**：请求与响应记审计日志；不得越权访问无权限 Workspace 数据。
- **验收**：Dashboard 智能分析与 Tracing AI 筛选调用同一鉴权体系。

---

### 4.6 安全与合规

#### F-701 客户端数据脱敏
- SDK 在上报前执行 `mask(input|output|metadata)`；默认不替换则原样上报。
- **验收**：开启 mask 后 UI 中对应字段显示 `[REDACTED]`，原始值不可从 API 恢复。

#### F-702 数据保留策略 TTL
- Workspace 设置中配置保留天数；定时任务物理删除过期 Observation 与 Trace。
- **验收**：过期数据在列表与详情均不可查，且不可通过 API 恢复。

#### F-703 RBAC 权限控制
- 角色矩阵覆盖：查看 Trace、导出、配置 Workspace、管理 API Key；Read-Only 不可改配置。
- **验收**：只读用户访问 AI 观测可浏览列表与详情，但无删除/配置入口。

#### F-704 API Key 管理
- 每 Workspace 至少一对 Public（上报）/ Secret（查询）Key；支持吊销后旧 Key 立即失效。
- **验收**：吊销后 SDK 上报返回 401；新 Key 上报成功。

## 5. 非功能需求 (NFR)

### 5.1 性能与延迟
- **服务端写入**：Ingestion API 需支持高并发写入，P99 响应时间 `< 200ms`。支持水平扩展以应对突发流量。
- **查询性能**：Trace 列表页及 Dashboard 聚合查询，在千万级数据量下，P95 响应时间 `< 2s`。

### 5.2 可用性与可靠性
- 系统整体可用性目标：99.9%。
- 数据持久性：99.999%。上报的数据一旦进入服务端 Ingestion 队列，必须保证最终一致性落盘，不允许丢失。

## 6. 数据埋点与运营指标 (内部监控)
作为平台产品，我们自身也需要监控平台的健康度：
1. **接入活跃度**：每日活跃 Workspace 数 (DAU Workspaces)、每日新增 Trace 数量。
2. **SDK 健康度**：各语言 SDK 的上报成功率、平均 Batch Size、网络超时率。
3. **存储成本**：单条 Trace 平均占用存储空间、总存储成本趋势。
4. **功能使用率**：Session 视图使用率、Masking 功能开启率、Cost Dashboard 访问量。

## 7. 后续演进规划 (Roadmap)
*注：本期 PRD 聚焦于核心的 Observability 闭环，以下功能将在后续版本迭代。*
- **Phase 2: 评估与测试 (Evaluation)**：支持在 Trace 上挂载 Evaluation 任务，通过 LLM-as-a-Judge 或人工打分，对 Trace 的 Output 进行质量评估，并展示在 UI 上。
- **Phase 3: Prompt 管理**：将 Trace 中的 Prompt 提取并版本化管理，支持在 UI 上直接修改 Prompt 并 A/B 测试，形成“追踪->分析->优化”的闭环。
- **Phase 4: 异常告警**：支持配置规则（如：某模型延迟连续 5 分钟 > 3s，或 成本突增 200%），通过 Webhook/邮件/钉钉触发告警。
